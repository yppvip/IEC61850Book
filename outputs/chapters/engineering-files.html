SCL 与工程交换文件

用文件类型理解工程交付、配置能力与系统集成边界

先建立正确的文件观念

SCL（Substation Configuration Language）是 XML 交换语言；SCD、ICD、CID、SSD、SED 和 IID 是以不同工程目的保存的 SCL 文件类型。它们不只是扩展名不同，而是表达的工程阶段、允许包含的信息和使用对象不同。

阅读原则：先判断文件类型和工程阶段，再解释其中的元素。一个 IED 的能力声明、全站的集成配置和装置可下载配置，不能互相替代。

文件类型速查

类型英文全称主要用途通常由谁使用
SSDSystem Specification Description系统规格与一次系统/功能需求的起点，不是某台 IED 的最终配置。设计、系统工程
ICDIED Capability Description描述 IED 支持的能力和可配置的数据模型、服务；不是已完成项目的装置配置。IED 厂商、系统集成
SCDSubstation Configuration Description全站系统配置，汇集一次系统、通信、IED 和类型模板，并表达系统集成结果。系统集成、工程交付
CIDConfigured IED Description面向单台 IED 的已配置描述，常作为下装或导入设备配置的依据。IED 工程、调试
SEDSystem Exchange Description跨项目或跨系统边界交换系统相关配置的描述。系统之间的工程交换
IIDInstantiated IED Description已实例化 IED 描述，服务于 IED 实例与系统工程交换的场景。IED/系统工程工具
工程注意：各工具对导入、导出以及各文件类型的支持范围可能不同。应以项目规范、工具说明和 IED 一致性声明确定实际交换流程。

典型工程关系

系统需求 / 一次系统设计 → SSD；IED 能力输入 → ICD；系统集成与通信/订阅配置 → SCD；单台 IED 已配置描述 → CID；跨系统/实例工程交换 → SED / IID。

这是理解用的常见逻辑，不表示所有项目都严格按这一文件流转，也不表示文件间可以仅凭后缀机械转换。

SCL 根元素与 Header

对象/属性标准语义工程阅读要点
SCL整个 SCL 文档的根元素。观察默认命名空间、包含哪些顶层块，以及是否有额外私有命名空间。
Header文件头与工程文件标识信息。id 标识文件；version/revision 是工程文件修订信息，不等同于 IEC 标准 Edition；toolID 可指示生成工具。
History/Hitem可记录修改历史。when、who、what、why 适合审计；可靠程度仍取决于工具与工程管理流程。

命名空间、版本与 Private

标准命名空间

根元素上的默认 xmlns 用于标识 SCL 词汇表。工程文件可能保留较早命名空间 URI；仅凭该 URI 不能推断 IED 的全部模型细节或 Edition 支持情况。

工程文件版本

Header@version 和 Header@revision 是文件自身的工程版本，便于交付和变更管理；它们不是“Edition 2”或“Edition 2.1”的替代写法。

Private 与扩展命名空间

工程/厂商扩展 Private 是容纳私有信息的扩展位置；带前缀的非 IEC 命名空间元素也可能是厂商或工具扩展。应在转换、合并或校验时谨慎保留其内容；没有来源厂商文档时，只能确认其扩展性质，不能由名称推断业务含义。

文件级排查清单

  1. 确认文件类型、交付用途、来源工具和目标工具。
  2. 检查 Header 的标识、修订与历史信息，不把它当作 IEC Edition。
  3. 检查 IED 名、AccessPoint 名、通信引用和类型模板引用的关联一致性。
  4. 记录默认命名空间和所有私有命名空间；迁移时先评估工具兼容性。
  5. 不要把 SCD 视为保护定值、联锁设计或安全策略的唯一依据；结合项目图纸、设备手册和调试记录核对。
