IED 与数据模型

从设备接入点走到数据属性，并理解控制、设置、质量和时标

IED 内部层级

IED → AccessPoint → Server → LDevice → LN0 / LN → DO → DA / BDA；LN0/LN 实例通过 lnType 指向 DataTypeTemplates 中的 LNodeType，再经 DOType、DAType、EnumType 追溯结构与枚举。
对象含义关键阅读点
IED智能电子设备的工程模型。name 是引用该设备的关键工程标识；type 或名称只能辅助识别设备角色。
AccessPointIED 的通信接入点。name 要与 Communication/ConnectedAP@apName 对应。
Server逻辑服务器容器。常承载一个或多个 LDevice；“Server”是模型概念，不等于独立物理服务器。
LDevice逻辑设备（LD）。inst 区分实例；内部必须理解为一个公共 LN0 加多个功能 LN。
LN0逻辑设备的零号逻辑节点。数据集、报告控制块、GOOSE/采样值控制块、设置组控制等常位于此处。
LN按功能组织数据的逻辑节点。lnClass 是类别，prefix 和 inst 用于实例区分，lnType 关联类型模板。

数据对象、数据属性与实例配置

层级含义例子与注意事项
DO数据对象，表达一类功能数据。例如位置、测量、控制相关对象；其内部结构由 DOType 定义。
DA数据属性，是可访问的数据基本成员或结构入口。常见属性包括状态值、质量、时标、设定值等；用途由 fc 进一步限定。
BDA基本数据属性，属于结构化 DA 的内部成员。用于展开复合数据类型；沿 DAType 继续追溯。
DOI/SDI/DAI在 SCL 中对 DO、子 DO、DA 的实例化配置。Val 是配置值的一种承载位置，不能自动等同于所有运行时实时值。
追溯路径：看到一个 DAI/Val 时，应依次找实例所属 LN 的 lnType、相应 LNodeType 的 DO、对应 DOType 的 DA，再判断是否关联 DAType 或 EnumType。不要只凭值文本猜测其编码含义。

功能约束（FC）

fc 标识数据属性按服务语义划分的用途，不是数据类型，也不是 XML 文件属性的通用枚举。常见功能约束如下：

FC常见含义工程注意
ST状态用于状态类数据；通常会结合质量与时标阅读。
MX测量用于测量类数据；单位、倍率和质量应一并核对。
CO控制与控制服务和控制模型相关；不是“已执行控制”的结果值。
SP可设定参数可配置参数的服务语义；权限和实际生效方式依 IED 实现。
SG / SE设置组 / 可编辑设置与设置组编辑、切换和确认流程有关，不能把编辑值直接当作当前生效值。
CF配置配置数据；修改通常涉及工程或装置配置流程。
DC描述描述性信息。

控制模型与控制对象

控制数据对象常包含 ctlModel 等属性；实际控制路径涉及客户端服务、控制模型、操作来源、闭锁、模式和 IED 能力。ctlModel 是数据属性值，不能与 SCL XML 属性混淆。

常见控制模型值含义
status-only仅有状态，不提供该对象的控制服务。
direct-with-normal-security直接操作、普通安全模型；仍受闭锁、权限、模式等约束。
sbo-with-enhanced-security先选择后执行，并采用增强安全相关机制；具体交互必须匹配 IED 支持能力。
不可只看 ctlModel 放行遥控。遥控工程正确性还需核对 Oper/SBO 服务、CSWI/CILO 功能、权限、闭锁、遥控对象及项目安全策略。

设置、质量与时标

概念含义阅读要点
设置组让一组可设定参数按组管理的机制。区分当前有效设置、可编辑设置和工程中定义的初始值；切换流程由控制与 IED 能力约束。
q（Quality）数据质量信息。值本身与质量必须同时判断；有效性、来源、测试/阻塞、溢出等质量语义应按对应数据类型和设备实现解释。
t（Timestamp）数据时标。表示与数据相关的时间信息；准确度、同步来源和事件顺序能力不能只由存在 t 推断。

IED 模型排查清单

  1. IED、AP、Server、LD、LN 的名称和实例标识是否与通信和一次系统引用一致。
  2. 每个 LN@lnType 是否能解析到类型模板；DO/DA 路径和枚举值是否沿模板追溯。
  3. 数据集的 FCDA 是否定位到正确的 LN、DO、DA 和 FC。
  4. 控制对象是否同时核对控制模型、闭锁、操作路径和权限。
  5. 测量/状态数据是否连同 q、t、单位及工程缩放信息一起理解。
