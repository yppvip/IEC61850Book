IEC 61850 报文解析

从 Wireshark 抓包走到每个字节：MMS、GOOSE、SV/SMV 的组织、编码和开发实现

先把报文放回正确层级

IEC 61850 的报文不是一种单独的线缆格式。抓包时先判断它走哪条承载路径，再解释业务字段。

报文常见承载抓包入口解析重点
GOOSEEthernet 二层多播，可带 802.1Q VLANeth.type == 0x88b8 或 gooseEthernet/VLAN、APPID、GOOSE PDU、stNum/sqNum、allData 与 SCL DataSet 对应关系。
SV / SMVEthernet 二层多播，可带 802.1Q VLANeth.type == 0x88ba 或 svAPPID、ASDU、svID、smpCnt、采样值序列、质量位与 MU 数据集映射。
MMS / ReportTCP/IP 上的 ISO-on-TCP、COTP、Session、Presentation、ACSE、MMStcp.port == 102 或 mmsTCP 重组、OSI 会话/表示层、ACSE 建链、MMS PDU、报告字段和 BER TLV。
阅读顺序：先确认帧类型和长度，再解释协议字段，最后才用 SCL 反查工程语义。不要拿 SCL 里的 APPID 或 DataSet 直接替代抓包证据。

抓包后的 Wireshark 基本动作

  1. 用 eth.type == 0x88b8 || eth.type == 0x88ba || tcp.port == 102 缩小 IEC 61850 相关流量。
  2. 打开 Packet Bytes 窗格，点击字段树中的任意字段，确认 Wireshark 高亮的字节范围。
  3. GOOSE/SV 先核对目的 MAC、VLAN、APPID、Length；MMS 先核对 TCP 是否有分段重组提示。
  4. 右键字段选择 Copy → Bytes → Hex Stream，可把当前字段的原始字节复制给解析器测试用例。
  5. 遇到 Malformed Packet，优先检查前一层 Length 是否越界、BER 长度是否不完整、TCP 分段是否未重组。
常用显示过滤器
GOOSE：eth.type == 0x88b8
SV：eth.type == 0x88ba
MMS：tcp.port == 102 || mms
指定 APPID：goose.appid == 0x1001 或 sv.appid == 0x4001
指定 VLAN：vlan.id == 10
观察 GOOSE 状态变化：goose.stNum || goose.sqNum
观察 SV 采样计数：sv.smpCnt

字节编码基础

概念字节规则解析器注意点
网络字节序以太网类型、VLAN TCI、APPID、Length、端口等多字节整数通常按大端读取。0x88 0xb8 读作 0x88B8，不是 0xB888。
Ethernet Header目的 MAC 6 字节、源 MAC 6 字节、Type/Length 2 字节；若为 0x8100，后续 4 字节是 VLAN Tag。带 VLAN 时真实 EtherType 在 VLAN Tag 之后，偏移会后移 4 字节。
BER TLVTag 标识字段，Length 标识值长度，Value 是字段内容。短长度 0x00..0x7f 直接表示长度；长长度 0x81 nn、0x82 nn nn 表示后续长度字节数。解析时永远先校验 offset + header + length <= buffer.length。
BOOLEANBER 常见值长度 1 字节，00 为 false，非零为 true。不要把布尔字段当字符串或整数枚举显示。
BIT STRINGValue 第 1 字节是“未使用位数”，后续才是实际位图。质量位、触发位等位串要先跳过 unused-bits 字节。
VisibleString / OCTET STRING字符串按字节序列解释；八位串只表示原始字节，不保证可打印。GOOSE 的 gocbRef 可显示为字符串，SV 的 sample 是二进制采样序列。

从 GOOSE 字节回到 SCL

  1. 用 APPID、目的 MAC、VLAN 在 Communication/SubNetwork/ConnectedAP/GSE/Address 里找到候选发布流。
  2. 用 gocbRef 定位发送 IED 的 LN0/GSEControl。
  3. 用 datSet 找到 DataSet，按 FCDA 顺序解释 allData 的第 1、2、3 项。
  4. 用 stNum 判断状态是否变化，用 sqNum 判断重发和丢包迹象，用 confRev 判断发送/接收配置是否同版。

SV 的 sample 没有自带通道名。通道顺序来自发送侧 SampledValueControl@datSet 指向的数据集。开发解析器时必须把 svID/APPID 与 SCL 中的 FCDA 顺序绑定，否则只能得到“第 N 个值”，不能知道它是 A 相电流还是 B 相电压。

MMS / 报告：从 TCP 流到 BER TLV

MMS 不像 GOOSE/SV 那样直接从二层 EtherType 进入业务字段。Wireshark 中常见层级是：

Ethernet → IP → TCP/102 → TPKT → COTP → ISO Session → ISO Presentation → ACSE → MMS
层开发解析关注点Wireshark 核查
TCP同一个 MMS PDU 可能跨多个 TCP segment；解析器必须先完成流重组。查看是否出现 TCP segment of a reassembled PDU。
TPKT常见头部以 03 00 开始，后 2 字节表示 TPKT 总长度。长度不够时，下层 COTP/MMS 会被截断。
COTP / Session / Presentation负责 OSI 传输、会话和表示上下文，不是 IEC 61850 数据对象本身。建链阶段会看到 Connect、Accept、ACSE AARQ/AARE。
MMS业务 PDU，例如 Initiate、Read、Write、InformationReport、ConfirmedResponse。展开 MMS 节点，再用 BER 字节确认字段来源。

解析器实现分层

Packet
  EthernetReader：目的 MAC / 源 MAC / VLAN / EtherType
  GooseReader：APPID / Length / Reserved / GOOSE APDU
  SvReader：APPID / Length / Reserved / SV APDU
  TcpStreamReassembler：按连接重组 TCP 字节流
  TpktCotpReader：TPKT / COTP / Session / Presentation
  BerReader：readTag / readLength / readValue / bounds check
  SclResolver：APPID、gocbRef、svID、DataSet、FCDA 顺序反查
  Diagnostics：长度越界、未知标签、confRev 不一致、丢帧、字段缺失
实现点建议
偏移推进每读一个字段返回 {tag, length, valueStart, valueEnd, nextOffset}，不要在多个函数里重复手算。
长度校验任何 Length 都必须先确认不会越过当前父字段边界和报文总长度。
未知字段保留原始 TLV 和偏移，标记为 unknown；不要因为未知标签直接丢弃整个 PDU。
SCL 反查GOOSE 用 APPID + gocbRef + datSet，SV 用 APPID + svID + SampledValueControl，MMS 用对象引用和报告控制块。
测试用例为每个层级保留最小十六进制样例、长度错误样例、未知标签样例、截断样例和真实脱敏样例。

现象到字段的排障矩阵

现象先看字段判断路径
抓不到 GOOSE/SVEtherType、目的 MAC、VLAN、镜像口方向先证明二层帧存在，再看 APPID 和业务字段。
订阅不生效APPID、gocbRef/svID、confRev、DataSet 顺序抓包字段与 SCL 的 GSE/SMV、控制块和 ExtRef 同时核对。
GOOSE 频繁超时timeAllowedToLive、sqNum、到达间隔比较报文间隔是否超过允许时间，再查网络丢帧、VLAN 优先级和发送装置状态。
GOOSE 状态未变化stNum、sqNum、allDatasqNum 增而 stNum 不增，多半是同状态重发；要看 allData 是否真的变化。
SV 丢帧或保护无量smpCnt、到达周期、sample 长度、质量位先看采样计数是否连续，再按 SCL DataSet 顺序解释通道与质量。
MMS 报告无数据TCP/102、ACSE 建链、ReportControl、InformationReport、invokeID先证明连接和使能流程成立，再看 RCB、触发项、权限和报告内容。
Wireshark 显示 malformed上一层 Length、BER Length、TCP 重组多数先从长度和截断查起；不要直接断定装置业务错误。
