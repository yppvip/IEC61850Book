现象到排障路径库

从可见症状出发，沿 SCL 引用链、网络与设备能力逐段收敛

安全与使用边界

本库用于定位思路，不提供绕过闭锁、权限、测试措施或现场安全规程的操作方法。控制、保护和过程层故障应先按项目流程确认安全状态；任何 SCD 修改、IED 下装、网络参数调整或现场试验均须经授权、审批和回退准备。

通用记录模板：症状出现时间、受影响 IED/AP/LD/LN、服务类型、工程版本、最近变更、抓包或事件证据、已核对的引用链。没有这些上下文时，不应只靠某个 XML 标签下结论。

GOOSE 收不到 / 订阅无效

症状条目 接收 IED 的输入不变化、告警提示订阅无效，或抓包可见/不可见发送报文。

按顺序核对目的
1. 接收侧 Inputs/ExtRef记录来源 IED、LD、LN、DO/DA、服务类型和工程关联；确认接收 LN/LD 实例本身存在。
2. 发送 DataSet / FCDA确认 ExtRef 对应数据确在发送侧数据集内，且 FC、DO/DA、实例标识一致。
3. GSEControl 与 Communication/GSE核对 datSet、cbName、ldInst 的关联，再核对 MAC、APPID、VLAN、子网与端口设计。
4. 装置与网络证据在授权条件下检查发送控制块是否使能、接收订阅状态、报文抓取、交换机 VLAN/组播/冗余路径。
常见误判：仅 MAC 相同不代表数据集或订阅正确；confRev 不是网络时延；SCD 中存在 ExtRef 也不等于现场装置已加载并接受该订阅。

SV 异常 / 保护无量

症状条目 保护或测控无电流/电压、数值异常、相序异常，或订阅状态提示采样值不可用。

按顺序核对目的
1. MU 发送模型从 SampledValueControl@datSet 到 DataSet/FCDA，确认数据通道、LN、DO/DA 与预期电流/电压量一致。
2. SMV 网络关联核对 Communication/SMV 的 LD、控制块名、MAC、APPID、VLAN 与网络设计。
3. 接收映射与量纲核对接收 ExtRef、相别/通道顺序、采样率、额定值/倍率和 IED 支持能力。
4. 一次与时间条件依项目程序检查互感器/电子式互感器链路、接线极性、时间同步和设备诊断；不要以单一 SCD 字段判断真实采样质量。
常见误判：SV 与 GOOSE 不是同一种服务；网络地址匹配也不能证明相序、通道和倍率正确；“有数据”不等于质量 q 合格。

报告无数据

症状条目 MMS 客户端已连接但未收到报告、报告不触发，或缓冲报告与预期不符。

按顺序核对目的
1. 客户端/连接状态先确认访问目标 AP、网络路径、客户端权限及所使用的报告实例；连接存在不等于 RCB 已使能。
2. ReportControl@datSet确认报告引用的数据集存在，FCDA 可以解析到目标数据，且实际数据具备变化或完整性条件。
3. TrgOps / OptFields检查数据变化、品质变化、周期触发及所需附加字段；按 IED 支持能力解释。
4. 缓冲与服务能力检查 buffered 语义、缓冲容量、使能状态和 IED 服务声明；必要时按厂家工具日志定位。
常见误判：buffered="true" 不保证无限历史；Report 不是 GOOSE 多播；SCD 中的 ReportControl 不自动代表某客户端已完成使能。

遥控失败

症状条目 选择/执行失败、控制被拒绝、无反馈，或操作后设备未动作。

按顺序核对目的
1. 目标对象与控制模型从目标 LN/DO/DA 类型确认控制对象与 ctlModel，区分 status-only、direct、SBO 等模型及 IED 服务能力。
2. 操作服务与权限核对客户端使用的 Oper/SBO 路径、操作来源、权限、超时/选择状态及 IED 日志。
3. CSWI / CILO / XCBR 链追踪控制协调、闭锁、命令输出与断路器反馈，分开查看“命令接受”“输出执行”“一次反馈”。
4. 现场条件按项目流程确认远方/就地、检修/测试、闭锁、一次设备条件和安全许可；仅可在授权试验环境进一步操作。
常见误判：看到 ctlModel 或 SCD 中一个 Val 不代表遥控已获准；控制成功应分别验证服务响应、逻辑输出和一次设备反馈。

SCD 导入失败

症状条目 工具拒绝导入、合并报错、导入后对象缺失或通信/类型引用断开。

按顺序核对目的
1. 文件用途与工具边界确认输入到底是 SCD、ICD、CID、SSD、SED 还是 IID，记录来源/目标工具版本与项目要求的 SCL Edition。
2. XML 与命名空间检查 XML 编码、根元素、默认/私有命名空间和结构完整性；标准命名空间 URI 是标识，不是在线依赖。
3. 引用完整性检查 IED/AP、LD/LN、控制块/DataSet、Communication/GSE/SMV、lnType/DOType/DAType/EnumType 的目标是否存在。
4. 扩展与差异对 Private、私有命名空间和厂商扩展保持原样并查资料；不要为了通过导入而删除未知扩展。
常见误判：XML 语法通过不等于工程语义或工具兼容通过；Header 的 version/revision 也不等同于 IEC Edition。

枚举值不明

症状条目 看到 Val、整数或状态编码，但不知道业务含义，或不同工具显示不一致。

按顺序核对目的
1. 定位值的性质确认它是 DAI/Val 配置值、运行服务读数，还是 XML 属性/类型模板的枚举；三者不能混读。
2. 沿类型模板追溯从 LN@lnType → LNodeType → DOType → DA/DAType，确认该 DA 的 bType 和类型引用。
3. 检查 EnumType读取相应 EnumVal@ord 与文字的对应，确认它只适用于该枚举类型和数据属性上下文。
4. 保留扩展边界若类型或值来自 Private/私有命名空间，记录来源和厂商文档需求；没有资料时仅标为扩展，不猜业务含义。
输出结论时应写清：值所在元素/DA、配置或运行上下文、类型链、EnumType 编码，以及是否存在厂商扩展。不要只写“0/1 的含义”。
